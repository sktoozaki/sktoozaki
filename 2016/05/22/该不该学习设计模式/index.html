该不该学习设计模式

5月 22, 2016 发布在 技术

Java 中的设计模式一直以来都存在争议，GoF 总结了23种设计模式，赞成的人认为他们极大地简化了大型项目开发中的一些问题，反对的人则用 HelloWorld 举例将其一黑到底。

那么，到底该不该了解和学习这些设计模式，在刚接触 Java 的时候还真让人有点纠结。

在这里王垠把设计模式吐槽了个遍，考虑到他 PL 与函数式编程的背景和在 Google 纠结的实习过程，猜测应该会觉得各类设计模式将软件开发过度工程化了。

个人觉得，如果只是一个人的团队，代码完全可以随心所欲自由发挥，选择一门对程序员友好的语言，按自己习惯来就好。但是如果是多人协作且层次参差不齐的情况，设计模式的运用和掌握存在优点。类似于 Android 平台，大量地运用了Observer、Adapter等模式，即使是一个新人，在最初的阶段照葫芦画瓢地也能开发出较为健壮的代码来。

至于一些成熟的框架，在 Github 上的各类第三方库等更是如此，很多都是各类模式的经典应用。不可否认对于新手入门和学习代码来说，设计模式是一个较高的门槛。在这个阶段我觉得可以不必理会其中的细节，等到接触和开发的代码到了一定程度，有些模式只需稍加梳理，便能理解和应用。

就像古人从小便被教育背诵四书五经，每隔一段时间，对这些经典的理解都是不一样的。很多时候并不需要在一开始过度推敲细枝末节，建立起一个较为完善的整体概念之后，时间所积累下来的对事物的深入认识可以将一开始的困惑逐渐清除，然后再根据需要的程度对细节部分进行补全。
